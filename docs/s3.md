# S3 Backend Support

This document proposes a design for an Amazon S3 backend for GlassDB. It
analyzes the gaps between S3 and the current GCS-based assumptions, evaluates
several candidate approaches (with reasons for discarding most), and recommends
a design that requires a focused algorithmic change — replacing
version-based change detection with **writer-based change detection** — and
then permits a clean single-object S3 layout.

## Motivation

GlassDB currently supports Google Cloud Storage (GCS) as its only production
backend. The [README](../README.md) and [architecture doc](architecture.md)
note that adding S3 support "should be very easy" since S3 now supports
conditional writes. In practice, the gap is larger than it first appears: S3's
conditional primitives differ from GCS's in several important ways that affect
the backend interface and, more interestingly, expose an implicit coupling
between the transaction algorithm and GCS-specific behavior.

That coupling is worth removing on its own merits, independent of S3 support.

## S3 vs GCS: Capability Comparison

| Capability | GCS | S3 |
|---|---|---|
| Content version | Generation (int64, monotonic) | ETag (string, typically MD5 of content) |
| Metadata version | Metageneration (int64, monotonic) | None — ETag is the only version |
| Conditional write | `GenerationMatch`, `MetagenerationMatch` | `If-Match: <etag>` (since 2024) |
| Write-if-not-exists | `DoesNotExist` precondition | `If-None-Match: *` (since Nov 2024) |
| Conditional read | `GenerationNotMatch` | `If-None-Match: <etag>` |
| Atomic metadata-only update | `object.Update()` with metageneration conditions | **Not available** |
| Conditional delete | `GenerationMatch` + `MetagenerationMatch` | **Not available** |
| User metadata | Object metadata (key-value, mutable via Update) | User metadata (`x-amz-meta-*`, immutable after upload) |

The three missing capabilities — atomic metadata-only update, conditional
delete, and separate metadata versioning — together with the ETag-as-MD5
behavior, are the source of the difficulty.

## The Pivotal Observation

The transaction algorithm's central change-detection step is in `algo.go`:

```go
func sameVersionAfterLock(v storage.Version, meta backend.Metadata) bool {
    return v.EqualMetaContents(meta)
}
```

`storage.Version.EqualMetaContents` already has *two* code paths
(`internal/storage/version.go`):

```go
func (v Version) EqualMetaContents(m backend.Metadata) bool {
    if !v.B.IsNull() {
        return v.B.Contents == m.Version.Contents
    }
    writer, _ := lastWriterFromTags(m.Tags)
    return writer != nil && v.Writer.Equal(writer)
}
```

When the version is "local" (a value staged by an uncommitted transaction),
the algorithm compares **the `last-writer` tag**, not a storage version. When
the version is backend-resident, it compares the GCS `Generation`.

Both paths answer the same conceptual question — *"is this still the value the
caller read?"* — and both are correct. The duplication exists only because
GCS hands out a convenient monotonic counter that happens to coincide with
"data change". S3 doesn't.

The key insight: **we don't actually need the `Contents` version for change
detection.** The `last-writer` tag is already maintained on every committed
write, and it's already trusted by the algorithm for local versions. If the
algorithm uses it uniformly, the dependency on GCS-style dual versions
disappears, and S3 can be implemented with a single object per key.

This re-frames everything below.

## Backend Interface Requirements

The `backend.Backend` interface today exposes a two-component `Version`:

```go
type Version struct {
    Contents int64  // GCS: Generation
    Meta     int64  // GCS: Metageneration
}
```

Operations:

```go
ReadIfModified(ctx, path, version int64) (ReadReply, error)
SetTagsIf(ctx, path, expected Version, tags Tags) (Metadata, error)
WriteIf(ctx, path, value, expected Version, tags Tags) (Metadata, error)
WriteIfNotExists(ctx, path, value, tags Tags) (Metadata, error)
DeleteIf(ctx, path, expected Version) error
```

Key observations:

1. **`Contents` is overloaded.** It serves two purposes: a CAS token for
   conditional writes, and an identifier the algorithm uses to detect content
   changes. The pivotal observation above shows the second purpose can be
   served by the `last-writer` tag instead.

2. **`SetTagsIf` is the hot path for locking.** Every lock acquire and release
   goes through it. In GCS, this is a lightweight metadata-only PATCH. Any
   S3 implementation must keep its cost bounded.

3. **`Version` is `int64`-typed.** S3 ETags are strings. This is a type
   mismatch that must be resolved regardless of which design wins.

## Discarded Approaches

### 1. Embed All Tags in Object Content

**Idea:** Wrap all S3 objects in an envelope: `{tags, data}`. Every operation
(including `SetTagsIf`) rewrites the full envelope. Tags are stored in the
content rather than S3 user metadata.

**Why it fails:**

- **`GetMetadata` requires full object download.** GCS serves metadata via a
  HEAD request (~30 ms). With embedded tags, S3 would need to download the
  full object to extract tags, making metadata reads proportional to data
  size. The read-only-transaction fast path is dominated by `GetMetadata`,
  so this is a serious regression.

- **Non-standard object format.** S3 objects are no longer raw user data.
  External tools can't inspect or manipulate them.

### 2. CopyObject-to-Self for Metadata Updates

**Idea:** Implement `SetTagsIf` as a CopyObject with the source and destination
being the same key, using `x-amz-copy-source-if-match` for the condition and
`MetadataDirective: REPLACE` to update metadata.

**Why it fails:**

- **ETag doesn't change.** For standard S3 objects, the ETag is the MD5 of
  the content. CopyObject-to-self doesn't change the content, so the ETag
  stays the same. Two concurrent `SetTagsIf` calls with the same expected
  ETag both pass the `If-Match` check and both succeed, but only one's
  metadata persists. CAS semantics are violated and lock state can be lost.

- **No version progression.** Without an ETag change, subsequent conditional
  operations reuse the same token, providing no protection against later
  concurrent modifications.

### 3. ETag-to-Counter Mapping

**Idea:** Keep `backend.Version` as `int64`. Maintain an in-memory map from
ETag strings to monotonically increasing counters inside the S3 backend.

**Why it fails:**

- **Unbounded memory growth.** Every ETag ever seen is retained.
- **Per-process.** Counters are not stable across clients or restarts.
- **Brittle.** Debug tools and logs see meaningless counters instead of
  real ETags.

### 4. Two-Object Model (Data + Meta Sidecar)

**Idea:** Each logical key is stored as two S3 objects:

```
<path>          → user value (raw bytes), ETag → Version.Contents
<path>.__meta   → serialized tags (JSON),  ETag → Version.Meta
```

`SetTagsIf` updates only the meta object; data writes update only the data
object. This preserves the dual-version semantics of GCS and keeps the
transaction algorithm unchanged.

This is the obvious naive port of GCS semantics, and it works. It is not
proposed because the costs are concrete and the alternative below avoids
them:

- **Compound operations are no longer atomic.** `WriteIf` becomes two
  `PutObject` calls (data, then meta). If the data write succeeds and the
  meta write fails, the key has new content with stale lock/last-writer
  tags. Recovery is possible via the transaction log, but the inconsistency
  window is observable by other readers and complicates reasoning about the
  algorithm's invariants. GCS has no such window.

- **`DeleteIf` is not atomic.** Head-then-delete has a TOCTOU race. The
  window is small, but it exists.

- **`GetMetadata` doubles in latency.** GCS: 1 HEAD (~30 ms). S3 two-object:
  1 HEAD + 1 GET, even parallelized (~50 ms). `GetMetadata` is on the
  read-only fast path.

- **`SetTagsIf` triples in S3 call count.** The lock tags must be merged
  with existing tags (last-writer, etc.) under CAS, requiring a
  read-merge-write cycle on the meta object: GetObject → modify →
  PutObject with `If-Match`. ~100 ms vs GCS's ~50 ms.

- **Object count doubles.** Storage cost, `LIST` page size, `DELETE` cost,
  `PUT` call count all double. Orphan-meta cleanup becomes a recurring
  background concern.

- **Two CAS tokens to track per key**, perpetuating the `Contents`/`Meta`
  split in the interface even though only S3 forces it.

The doubling of objects and per-key calls is fundamental to this model and
not fixable without an algorithm change. So we make that change instead.

## Proposed Solution: Writer-Based Change Detection + Single Object

### Part 1: Algorithm change

Replace `Contents`-based change detection with `last-writer`-based change
detection, uniformly:

```go
// Before
func (v Version) EqualMetaContents(m backend.Metadata) bool {
    if !v.B.IsNull() {
        return v.B.Contents == m.Version.Contents
    }
    writer, _ := lastWriterFromTags(m.Tags)
    return writer != nil && v.Writer.Equal(writer)
}

// After
func (v Version) EqualMetaContents(m backend.Metadata) bool {
    writer, _ := lastWriterFromTags(m.Tags)
    return writersEqualOrBothNull(v.Writer, writer)
}
```

`Version.Writer` is populated whenever the metadata is read (from the
`last-writer` tag). Storage CAS tokens (`Version.B`) are still used by the
backend for conditional writes, but they are no longer compared in the
algorithm.

**Semantics:** "the value hasn't changed since I read it" becomes "the last
writer is still the one I observed". This is strictly more robust than the
ETag-based interpretation: two different transactions writing identical bytes
have different writer IDs, but could produce identical ETags. Today's
algorithm relies on GCS's monotonic Generation to avoid this; the new
algorithm relies on writer IDs, which is what was always logically meant.

**Edge case: keys with no `last-writer` tag** (pre-existing or
externally-created objects). The tag is absent. Both `v.Writer` and the
parsed writer are nil — `writersEqualOrBothNull` returns true. This is
correct: the algorithm only inspects change-since-read on keys it actually
read, and any key it read has either been written by GlassDB (writer set)
or didn't exist (the `NotFound` path, which has its own validation logic).

### Part 2: Single-object S3 layout

With the algorithm decoupled from ETags-as-content-version, the S3 backend
becomes a single object per key:

| Field | Storage |
|---|---|
| User value | Object body (with an 8-byte nonce prepended) |
| `lock-type`, `locked-by`, `last-writer` | S3 user metadata (`x-amz-meta-*`) |

**The nonce.** S3's ETag is the MD5 of the content. Without a nonce, two
sequential `SetTagsIf` calls that re-upload the same user data produce
identical ETags and concurrent calls can both pass `If-Match` (the
[CopyObject-to-self failure](#2-copyobject-to-self-for-metadata-updates)).
Prepending 8 random bytes on every `PutObject` guarantees a fresh ETag and
restores CAS semantics. The nonce is stripped on read.

### Operation Mapping

| Backend Method | S3 Implementation | Notes |
|---|---|---|
| `Read(path)` | `GetObject(path)` | Strip nonce; return body + parsed metadata |
| `GetMetadata(path)` | `HeadObject(path)` | Single call |
| `SetTagsIf(path, exp, tags)` | `GetObject(path)` → `PutObject(path, nonce∥body, new metadata, If-Match: exp)` | Re-upload; see cost discussion |
| `Write(path, val, tags)` | `PutObject(path, nonce∥val, metadata)` | Single atomic call |
| `WriteIf(path, val, exp, tags)` | `PutObject(path, nonce∥val, metadata, If-Match: exp)` | Single atomic call |
| `WriteIfNotExists(path, val, tags)` | `PutObject(path, nonce∥val, metadata, If-None-Match: *)` | Single atomic call |
| `Delete(path)` | `DeleteObject(path)` | Single atomic call |
| `DeleteIf(path, exp)` | `HeadObject(path)` to check ETag, then `DeleteObject(path)` | Race window; see below |
| `List(dirPath)` | `ListObjectsV2(prefix=dirPath)` | No filtering needed |
| `ReadIfModified(path, expectedWriter)` | `HeadObject(path)` → compare writer → `GetObject(path)` if changed | See cache section |

### Why It Works

The transaction algorithm's invariants:

- **`sameVersionAfterLock` after a read-locked key.** The locker writes new
  lock tags but preserves the `last-writer` tag (since `SetTagsIf` doesn't
  touch the writer). After locking, the writer tag is unchanged → check
  passes. **Correct.**

- **Detecting concurrent writes.** A concurrent committed write changes the
  `last-writer` tag (and the value). After we acquire the lock, the writer
  tag differs → check fails → retry. **Correct.**

- **`WriteIf` atomicity.** Single `PutObject` updates content + metadata
  atomically. There is no compound-write window. **Strictly better than the
  two-object model.**

- **`DeleteIf` atomicity.** Same race window as the two-object model: head
  then delete. Cannot be avoided on S3 without server-side support.
  **Equal.**

### The `SetTagsIf` Cost Trade-off

`SetTagsIf` on S3 must re-upload the object content (because changing the
ETag requires changing the content). The cost is therefore proportional to
value size:

| Value size | GCS `SetTagsIf` | S3 two-object `SetTagsIf` | S3 single-object `SetTagsIf` |
|---|---|---|---|
| 1 KB | ~50 ms | ~100 ms | ~80 ms (GET + PUT) |
| 100 KB | ~50 ms | ~100 ms | ~120 ms |
| 1 MB | ~50 ms | ~100 ms | ~400 ms |

For the documented target range of 1 KB – 1 MB, the single-object model is
competitive at the small end and noticeably slower at the large end. Lock
operations happen 2× per locked key per transaction (acquire + release),
so this matters for write-heavy, high-contention workloads on large values.

**Mitigation if it becomes a problem in practice.** A future optimization
can introduce a *value sidecar* for objects above a size threshold: store
large bodies in `<path>.__data` and keep `<path>` as a small object holding
only metadata + a pointer to the body. This keeps `SetTagsIf` cheap for
large values without affecting the algorithm or the common case. We do not
propose this now: the basic design should be measured first, and adding
two storage layouts up front is premature complexity.

### Cache Layer Adjustment

The cache (`internal/storage/local.go`) currently uses `B.Contents` to
detect when a cached value is outdated relative to cached metadata. With
the algorithm change, this becomes a `last-writer` comparison:

```go
// Before
if valVersion.Contents == metaVersion.Contents { ... }

// After
if writersEqual(valVersion.Writer, metaVersion.Writer) { ... }
```

`Global.Read`'s use of `backend.ReadIfModified` also needs adjustment.
Today, GCS uses `GenerationNotMatch` for a conditional GET that returns
304 when the value hasn't changed. With nonce-in-content, the S3 ETag
changes on every `SetTagsIf`, so an ETag-based `If-None-Match` is useless
for the cache.

The proposed `ReadIfModified` signature changes from `version int64` to
`expectedWriter data.TxID`. The implementations:

- **GCS:** `HeadObject` for the metadata, compare the writer tag; if
  equal, return `ErrPrecondition`; otherwise fetch the object. (One extra
  HEAD vs today's single conditional GET — minor regression that can be
  mitigated by caching the metadata HEAD response.)
- **S3:** Same pattern. One HEAD + (cache hit ? 0 : 1) GET.

This is the largest non-mechanical change in the algorithm impact.

## Backend Interface Changes

```go
// Before
type Version struct {
    Contents int64
    Meta     int64
}

// After: opaque CAS token, interpreted by the backend only.
type Version struct {
    Token string
}
func (v Version) IsNull() bool { return v.Token == "" }
```

The `Contents` / `Meta` split is removed. Backends format their token
however they like:

- **GCS:** `Token = fmt.Sprintf("%d/%d", gen, metagen)` — parsed back to
  `GenerationMatch` / `MetagenerationMatch` for conditional ops.
- **S3:** `Token = etag`. Used as `If-Match`.

`ReadIfModified` changes from `version int64` to `expectedWriter
data.TxID`, as described above.

`Metadata` continues to expose `Tags`, and the algorithm reads
`last-writer` directly from tags (already done today in the local-version
branch).

**Justification.** The project is alpha and interfaces are explicitly not
stable. The simpler, opaque-token interface is a better long-term shape:
it doesn't bake GCS's dual-version model into the abstraction, which means
future backends (Azure Blob, R2, in-memory test backends with arbitrary
semantics) implement only what they natively support.

## Algorithm Impact Summary

| Component | Change |
|---|---|
| `storage.Version.EqualMetaContents` / `EqualContents` | Use writer-tag comparison uniformly; drop the `B.Contents` branch |
| `local.go:isValueOutdated` / `isMetaOutdated` | Compare writer tags instead of `B.Contents` |
| `global.go:Read` | Use writer-based `ReadIfModified`; one extra HEAD per cache hit |
| `backend.Version` | `int64` Contents/Meta → opaque `Token string` |
| `backend.ReadIfModified` | `version int64` → `expectedWriter data.TxID` |
| GCS backend | Format/parse `Token`; implement writer-based `ReadIfModified` via HEAD + conditional GET |
| Transaction algorithm (`algo.go`) | **No structural change.** `sameVersionAfterLock` is unchanged; it routes through `EqualMetaContents`, which is the function that changes. |
| Locker (`locker.go`) | No change. `last-writer` tag is already managed correctly. |
| Transaction logs, GC, deadlock fallback, crash recovery | No change. |

The algorithm itself — S2PL with the parallel/serial fallback, the
read-only fast path, single-RMW shortcut, retry-with-locks-held, tx-log as
commit point, lock-takeover via CAS on tx log — is untouched.

## Limitations

| Limitation | Impact | Mitigation |
|---|---|---|
| `DeleteIf` is head-then-delete, not atomic | Small TOCTOU window during lock cleanup | Same as two-object model; existing algorithm recovery covers it |
| `SetTagsIf` re-uploads object content | O(value size) per lock op; expensive for large values | Acceptable for ≤100 KB; value sidecar optimization available for larger values |
| `ReadIfModified` cache hit needs an extra HEAD | ~30 ms added on cache hits | Acceptable; cache hit was already a sub-100ms operation |
| ETag-as-MD5 quirks (SSE-KMS, multipart) | ETag format varies | Opaque token treatment makes this irrelevant — we never inspect it |
| Keys without `last-writer` tag | External objects are indistinguishable from each other for change detection | Algorithm only inspects keys it has read; falls back cleanly when both sides are null |

## Future Considerations

- **Value sidecar for large objects.** If profiling shows `SetTagsIf` cost
  hurts large-value workloads, introduce `<path>.__data` for the body and
  keep `<path>` as a small metadata-bearing object. The algorithm and
  interface do not need to change; only the S3 backend.
- **Azure Blob Storage.** Azure has separate content + metadata versioning
  (similar to GCS). With the opaque token, the Azure backend can encode
  either or both into `Token`. No additional interface work.
- **Removing `Meta` from local-cache logic.** Once writer-based detection
  is the only path, the cache's separate-value-vs-metadata staleness
  reasoning may simplify further. Left as a follow-up.
- **Conditional delete on S3.** If S3 ever adds `If-Match` on `DeleteObject`,
  the `DeleteIf` race window closes automatically.
