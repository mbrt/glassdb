# S3 Backend Support

This document proposes a design for an Amazon S3 backend for GlassDB. It
analyzes the gaps between S3 and the current GCS-based assumptions, evaluates
several candidate approaches (with reasons for discarding most), and recommends
a two-object model that requires no changes to the transaction algorithm.

## Motivation

GlassDB currently supports Google Cloud Storage (GCS) as its only production
backend. The [README](../README.md) and [architecture doc](architecture.md)
note that adding S3 support "should be very easy" since S3 now supports
conditional writes. In practice, the gap is larger than it first appears: S3's
conditional primitives differ from GCS's in several important ways that affect
the backend interface and, potentially, the transaction algorithm.

## S3 vs GCS: Capability Comparison

| Capability | GCS | S3 |
|---|---|---|
| Content version | Generation (int64, monotonic) | ETag (string, typically MD5 of content) |
| Metadata version | Metageneration (int64, monotonic) | None â€” ETag is the only version |
| Conditional write | `GenerationMatch`, `MetagenerationMatch` | `If-Match: <etag>` (since 2024) |
| Write-if-not-exists | `DoesNotExist` precondition | `If-None-Match: *` (since Nov 2024) |
| Conditional read | `GenerationNotMatch` | `If-None-Match: <etag>` |
| Atomic metadata-only update | `object.Update()` with metageneration conditions | **Not available** |
| Conditional delete | `GenerationMatch` + `MetagenerationMatch` | **Not available** |
| User metadata | Object metadata (key-value, mutable via Update) | User metadata (`x-amz-meta-*`, immutable after upload) |

The three missing capabilities â€” atomic metadata-only update, conditional
delete, and separate metadata versioning â€” create the design challenges explored
below.

## Backend Interface Requirements

The `backend.Backend` interface defines these operations relevant to the
discussion:

```go
type Version struct {
    Contents int64  // GCS: Generation
    Meta     int64  // GCS: Metageneration
}

ReadIfModified(ctx, path, version int64) (ReadReply, error)
SetTagsIf(ctx, path, expected Version, tags Tags) (Metadata, error)
WriteIf(ctx, path, value, expected Version, tags Tags) (Metadata, error)
WriteIfNotExists(ctx, path, value, tags Tags) (Metadata, error)
DeleteIf(ctx, path, expected Version) error
```

Key observations:

1. **`Version` has two independent components.** `Contents` changes on data
   writes; `Meta` changes on metadata updates. The algorithm relies on this
   separation: `sameVersionAfterLock` (in `algo.go`) compares `Contents` to
   detect whether actual data changed, ignoring metadata-only changes from
   lock acquisition.

2. **`SetTagsIf` is the hot path for locking.** Every lock acquire and release
   goes through `SetTagsIf`. In GCS, this is a lightweight metadata-only PATCH.
   Any S3 implementation must keep this efficient.

3. **`Version` fields are `int64`.** S3 ETags are strings. This is a type
   mismatch that must be resolved.

## Discarded Solutions

### 1. Nonce-in-Content

**Idea:** Append a random 8-byte nonce to every object's content. On
`SetTagsIf`, re-upload the same user data with a new nonce, changing the ETag.
Store tags in S3 user metadata (`x-amz-meta-*`). Use `If-Match` on PutObject
for CAS semantics.

**Why it fails:**

- **Breaks `sameVersionAfterLock`.** After lock acquisition via `SetTagsIf`,
  the ETag changes (new nonce â†’ new content hash). The algorithm's
  `sameVersionAfterLock` check compares the read version's `Contents` against
  the current metadata's `Contents`. Since `Contents` changed (even though
  user data didn't), the check fails, and the transaction incorrectly retries.
  This creates an infinite retry loop.

- **Requires content re-upload.** `SetTagsIf` doesn't receive the current
  content as a parameter. The S3 backend would need to `GetObject` first (extra
  round-trip), then re-upload with the new nonce. For lock operations on large
  values (up to 1 MB), this means downloading and re-uploading megabytes of
  data just to change lock metadata.

- **Algorithm fix is invasive.** Fixing `sameVersionAfterLock` would require
  either propagating a "freshly acquired" flag from the locker to the algorithm,
  or updating `pathState.ReadVersion` after lock acquisition. Both touch the
  core transaction validation path.

### 2. Embed Tags in Object Content

**Idea:** Wrap all S3 objects in an envelope: `{nonce, tags, data}`. Every
operation (including `SetTagsIf`) rewrites the full envelope, producing a new
ETag. Tags are stored in the content rather than S3 user metadata.

**Why it fails:**

- **Same `sameVersionAfterLock` problem as nonce-in-content.** Any content
  modification changes the ETag, triggering false conflict detection.

- **`GetMetadata` requires full object download.** GCS serves metadata via a
  HEAD request (~30 ms). With embedded tags, S3 would need to download the
  full object to extract tags, making metadata reads proportional to data size.

- **Non-standard object format.** S3 objects are no longer raw user data.
  External tools can't inspect or manipulate them directly.

### 3. CopyObject-to-Self for Metadata Updates

**Idea:** Implement `SetTagsIf` as a CopyObject with the source and destination
being the same key, using `x-amz-copy-source-if-match` for the condition and
`MetadataDirective: REPLACE` to update metadata.

**Why it fails:**

- **ETag doesn't change.** For standard (non-encrypted, non-multipart) S3
  objects, the ETag is the MD5 of the content. CopyObject-to-self doesn't
  change the content, so the ETag stays the same. This means two concurrent
  `SetTagsIf` calls with the same expected version would both succeed, violating
  CAS semantics and potentially corrupting lock state.

- **No version progression.** Without an ETag change, there's no way to
  detect that metadata was updated. Subsequent conditional operations would use
  the same ETag, providing no protection against concurrent modifications.

### 4. ETag-to-Counter Mapping (Avoid Version Type Change)

**Idea:** Keep `backend.Version` as `int64`. Within the S3 backend, maintain
an in-memory map from ETag strings to monotonically increasing int64 counters.
Each unique ETag gets a new counter value. Convert back to ETag for S3 API
calls.

**Why it was discarded (as primary approach):**

- **Memory grows unboundedly.** Every ETag ever seen is stored. In a
  long-running process with many objects, this becomes significant.

- **Not needed with the proposed approach.** The two-object model (see below)
  naturally maps to two independent version strings. Changing `Version` to
  strings is cleaner.

- **Fragile.** The mapping is per-process. If debug tools or monitoring inspect
  version numbers, they see meaningless counters. Error messages become
  harder to interpret.

Note: this could work as a transitional strategy if the `int64` â†’ `string`
Version change is deferred.

### 5. Separate Lock Objects (Naive)

**Idea:** Store lock state in a companion object (`path.lock`) alongside the
data object. `SetTagsIf` writes to the lock object only.

**Why the naive version fails:**

- **`WriteIf` must be atomic across both objects.** The locker's
  `applyLockTags` sometimes calls `WriteIf` to write content AND update lock
  tags in a single operation. With separate objects, this becomes two writes.
  If the data write succeeds but the lock write fails, the system is in an
  inconsistent state: data was updated but lock metadata is stale.

- **Tag merge problem.** `SetTagsIf` in `locker.go` only writes the new lock
  tags (`lock-type`, `locked-by`). In GCS, this merges with existing metadata
  (e.g., `last-writer` is preserved). With a separate lock object, writing only
  the new tags would overwrite the entire lock object, losing `last-writer`.
  The S3 backend must explicitly read-merge-write.

However, a refined version of this approach â€” the two-object model â€” addresses
these issues and is the proposed solution.

## Proposed Solution: Two-Object Model

Each logical key is stored as two S3 objects:

| Object | Path | Contents | Purpose |
|--------|------|----------|---------|
| Data | `<path>` | User value (raw bytes) | Content reads and writes |
| Meta | `<path>.__meta` | Serialized tags (e.g. JSON) | Lock state and other metadata |

### Version Mapping

```
Version.Contents = data object ETag   (changes only on content writes)
Version.Meta     = meta object ETag   (changes only on tag updates)
```

These change **independently**, preserving the same semantics as GCS's
Generation/Metageneration. This is what makes the approach work: the
transaction algorithm sees the same version behavior it was designed for.

### Operation Mapping

| Backend Method | S3 Implementation | Notes |
|---|---|---|
| `Read(path)` | `GetObject(path)` | Returns user data + data ETag as `Version.Contents` |
| `ReadIfModified(path, v)` | `GetObject(path, If-None-Match: v)` | 304 â†’ `ErrPrecondition` |
| `GetMetadata(path)` | `HeadObject(path)` + `GetObject(path.__meta)` | Merge data ETag + meta tags |
| `SetTagsIf(path, exp, tags)` | `PutObject(path.__meta, tags, If-Match: exp.Meta)` | Data object untouched |
| `Write(path, val, tags)` | `PutObject(path, val)` then `PutObject(path.__meta, tags)` | Two calls |
| `WriteIf(path, val, exp, tags)` | `PutObject(path, val, If-Match: exp.Contents)` then `PutObject(path.__meta, tags)` | Conditional on data ETag |
| `WriteIfNotExists(path, val, tags)` | `PutObject(path, val, If-None-Match: *)` then `PutObject(path.__meta, tags)` | Conditional on data absence |
| `Delete(path)` | `DeleteObject(path)` + `DeleteObject(path.__meta)` | Best-effort on meta |
| `DeleteIf(path, exp)` | `HeadObject(path)` check ETag, then `DeleteObject` both | Race window; see below |
| `List(dirPath)` | `ListObjectsV2(prefix=dirPath)` | Filter out `.__meta` suffixes |

### Why `sameVersionAfterLock` Works

The critical validation step in `algo.go`:

```go
func sameVersionAfterLock(v storage.Version, meta backend.Metadata) bool {
    return v.EqualMetaContents(meta)
}
```

This checks whether `ReadVersion.Contents == metadata.Version.Contents`. With
the two-object model:

1. Transaction reads key â†’ `Version.Contents` = data ETag `D1`.
2. Lock acquisition: `SetTagsIf` writes to `path.__meta` only. Data object
   is untouched â†’ data ETag stays `D1`.
3. After locking: `GetMetadata` returns `Version.Contents` = `D1`.
4. Check: `D1 == D1` â†’ **passes correctly**.

The algorithm sees no version change because the data object genuinely didn't
change.

### Compound Operation Atomicity

Operations like `WriteIf` involve two S3 calls (data write + meta write). If
the data write succeeds but the meta write fails:

- The data object has a new ETag, but the meta object has stale lock state.
- The caller receives an error and the transaction fails.
- On retry, the algorithm re-reads and sees the new data version. The locker
  detects the stale lock state and resolves it through the normal recovery
  mechanism (inspecting transaction logs, aborting stale locks).

Mitigation: the S3 backend should retry the meta write a few times before
returning an error. Since the data write already succeeded, the lock is
effectively held; the meta write just needs to persist the new state.

### `DeleteIf` Race Window

S3 does not support conditional delete. The implementation is:

1. `HeadObject(path)` â†’ check ETag matches `expected.Contents`.
2. `HeadObject(path.__meta)` â†’ check ETag matches `expected.Meta`.
3. If both match: `DeleteObject(path)` + `DeleteObject(path.__meta)`.

Between steps 1â€“2 and step 3, another transaction could modify the object. In
the worst case, a concurrent write is deleted. However:

- `DeleteIf` is only used in `storage/locker.go` during lock cleanup
  (unlocking after aborted creates, deleting committed values marked for
  deletion).
- The algorithm's retry and recovery mechanisms handle unexpected failures.
- The race window is small in practice (milliseconds).

### `GetMetadata` Cost

GCS serves `GetMetadata` via a single HEAD request (~30 ms). The two-object
model requires two S3 calls:

1. `HeadObject(path)` â€” data ETag (~30 ms).
2. `GetObject(path.__meta)` â€” lock tags (~50 ms).

These can be parallelized, keeping latency around 50 ms. The meta object is
small (< 200 bytes), so bandwidth is negligible.

### Tag Merge on `SetTagsIf`

The `locker.go:applyLockTags` function writes only the new lock tags
(`lock-type`, `locked-by`). On GCS, this merges with existing metadata. On S3,
writing the meta object replaces its entire content.

The S3 backend must read the current meta object, merge the new tags, and write
back. This is safe because `SetTagsIf` already uses `If-Match` on the meta
ETag â€” if the meta object changed between read and write, the condition fails
and the caller retries.

Implementation: the meta object stores a JSON map of all tags. On `SetTagsIf`:

1. `GetObject(path.__meta)` â†’ deserialize current tags.
2. Merge new tags into existing map.
3. `PutObject(path.__meta, serialized, If-Match: expected.Meta)`.

## Backend Interface Changes

### Version Type: `int64` â†’ `string`

S3 ETags are strings like `"d41d8cd98f00b204e9800998ecf8427e"`. The current
`backend.Version` uses `int64`, which cannot represent ETags.

Proposed change:

```go
// Before
type Version struct {
    Contents int64
    Meta     int64
}

// After
type Version struct {
    Contents string
    Meta     string
}
```

`ReadIfModified` similarly changes from `version int64` to `version string`.
`Version.IsNull()` checks for empty string instead of zero.

**Justification:** The project is alpha and interfaces are explicitly not
stable. Strings are the correct long-term representation: they accommodate both
GCS's numeric generations (formatted as decimal strings) and S3's hex ETags.
All version comparisons in the codebase use `==` equality, which works
identically for strings.

**Impact:** ~15 files across `backend/`, `internal/storage/`,
`internal/trans/`, and `tx.go`. Changes are mechanical (type substitution +
format/parse conversions in GCS backend).

## Algorithm Impact

**No changes required.** The transaction algorithm â€” locking protocol, commit
sequence, validation logic, retry mechanisms, deadlock handling, and crash
recovery â€” all work unchanged with the two-object model. The backend
abstraction properly insulates the algorithm from storage specifics.

The cache layer (`local.go`'s `isValueOutdated` / `isMetaOutdated`) also works
correctly because `Contents` and `Meta` change independently, exactly as they
do on GCS.

## Limitations

| Limitation | Impact | Mitigation |
|---|---|---|
| Compound writes not atomic | Brief inconsistency between data + meta objects | Retry meta write; algorithm recovery handles stale state |
| `DeleteIf` race window | Possible stale delete during lock cleanup | Small window; retry logic recovers |
| `GetMetadata` = 2 calls | ~50 ms vs ~30 ms on GCS | Parallelize the two calls |
| `SetTagsIf` = read-merge-write | ~100 ms vs ~50 ms on GCS (3 S3 calls) | Acceptable; lock operations are not the bottleneck |
| Double object count | 2Ă— storage, `List` needs filtering | Meta objects are tiny; filter is O(n) |
| Orphaned meta objects | Delete partially fails | Periodic garbage collection or background cleanup |

## Future Considerations

- **S3 may add new conditional primitives.** If S3 adds conditional delete or
  metadata-only conditional update, the implementation can be simplified without
  changing the interface.
- **Azure Blob Storage.** Azure has separate content + metadata versioning
  (similar to GCS), so the two-object model would not be needed there. The
  string-based `Version` type would still be required.
- **Performance optimization.** If profiling shows `SetTagsIf` latency is a
  bottleneck, the meta read in the read-merge-write cycle could be avoided by
  having the caller pass the full tag set (not just the delta). This would
  require a minor interface change.
