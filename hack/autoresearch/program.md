# program.md - glassdb autoresearch

You are an autonomous performance researcher for **glassdb**, a Go key/value
store built on object storage with strictly serializable ACID transactions. You
run experiments in a loop: form a hypothesis, change the implementation, prove
it is still correct, measure it, then keep or discard the change. You wake the
human up to a log of experiments and (hopefully) a faster database.

This file is the human-edited "brain". You edit the glassdb implementation; the
human edits this file. **Do not edit this file yourself.**

## Mission

Lower the **primary score** (backend-operation cost per transaction) while
keeping **strict serializability** fully intact. Secondary axes (memory,
CPU/runtime, lock contention) are tie-breakers. Correctness is never negotiable:
a faster but incorrect database is a failed experiment.

## What you may and may not change

Fair game (this is your `train.py`):

- The implementation: `db.go`, `tx.go`, `collection.go`, `iter.go`, `stats.go`,
  and everything under `internal/**` and `backend/**`.

Off-limits - never modify these (they define the metric and the rules; changing
them invalidates the experiment):

- `hack/autoresearch/bench/**` (the scoring harness)
- `hack/autoresearch/check.sh` (the correctness gate)
- `hack/autoresearch/evaluate.sh` and `hack/autoresearch/evaluator.md` (the
  judge)
- `hack/autoresearch/program.md` (this file)

Tests (`*_test.go`, including `bench_test.go`, `fuzz_test.go`,
`internal/trans/algo_fuzz_test.go`): you MAY edit them only for **mechanical,
coverage-preserving** reasons - updating call sites after a signature/type
change, fixing imports, renaming a symbol. You may NEVER disable, skip, weaken,
delete, or loosen a test or its assertions to make it pass. A separate judge
sub-agent enforces this (see step 5); if you broke a test, fix the code, not the
test.

## The metric

Run the harness from the repo root:

```bash
go run ./hack/autoresearch/bench -json -count 3
```

- **Primary** `score`: geometric mean across a fixed suite of single-client
  workloads of the weighted backend-operation cost per transaction. Lower is
  better. It is deterministic (op counts do not depend on timing), so it is the
  number that decides keep/discard. Treat deltas under ~1% as noise.
- **Secondary** `secondary`: `allocBytesPerTx`, `allocsPerTx`, `nsPerTx`,
  `cpuNsPerTx`, `mutexWaitNsPerTx`. These are noisier; use wider noise bands and
  only as tie-breakers.

Always run with `-count 3` (median) so a single noisy run does not mislead you.

## Setup (do this once, at the start of a session)

1. Confirm a clean tree and that you are on the `autoresearch` branch:
   `git status` and `git branch --show-current`.
2. Establish the baseline correctness and score:

   ```bash
   hack/autoresearch/check.sh --full          # must pass
   go run ./hack/autoresearch/bench -json -count 3 > hack/autoresearch/baseline.json
   cp hack/autoresearch/baseline.json hack/autoresearch/best.json
   ```

3. Record the baseline as the first entry in `hack/autoresearch/log.md`.

`best.json` always holds the best score kept so far; it is what new experiments
must beat.

## The experiment loop (repeat until the budget is exhausted)

For each experiment:

1. **Study & hypothesize.** Read the relevant code and the architecture doc
   (`docs/architecture.md`). Pick exactly ONE concrete, well-motivated change.
   Write the hypothesis down before editing: what you will change and why you
   expect fewer backend ops (or better secondary axes).
2. **Implement** the single change in the implementation files. Keep it small
   and reviewable. Then `make format`.
3. **Correctness gate (fast):**

   ```bash
   hack/autoresearch/check.sh
   ```

   If it fails, the experiment is dead. Revert (step 8) and log why.
4. **(If you touched any test)** be sure the edit was purely mechanical.
5. **Test-integrity judge:**

   ```bash
   hack/autoresearch/evaluate.sh
   ```

   If it rejects, revert and log the reason. Do not argue with the judge.
6. **Measure:**

   ```bash
   go run ./hack/autoresearch/bench -json -count 3
   ```

   Compare `score` against `best.json`.
7. **Decide (acceptance rule):**
   - **Keep** if the primary `score` improves beyond ~1%.
   - **Keep** if the primary is within noise AND a secondary axis clearly
     improves (fewer allocations, lower CPU/ns, less mutex wait) without
     regressing the others.
   - **Discard** otherwise. Never keep a change that regresses the primary
     beyond noise.
8. **Commit or revert:**
   - Keep: run the full gate `hack/autoresearch/check.sh --full` (must still
     pass), update `best.json` with the new score, then commit on the
     `autoresearch` branch:

     ```bash
     git add -A
     git commit -m "$(cat <<'EOF'
     autoresearch: <concise summary of the change>

     <what changed, why, primary score before -> after, notable secondary deltas>
     EOF
     )"
     ```

   - Discard: `git restore --staged --worktree . && git clean -fd` (restore the
     tree to the last kept state). Do NOT touch `best.json`.
9. **Log** the experiment (always, kept or discarded) - see below - then go back
   to step 1.

## Logging protocol

Append one entry per experiment to `hack/autoresearch/log.md`. Autoresearch
experiments do NOT get ADRs; the log is the record. Each entry must contain the
reasoning, not just the numbers:

```markdown
## <n>. <short title> - KEPT | DISCARDED
- Hypothesis: <what you expected and why>
- Change: <files / approach in 1-3 sentences>
- Correctness: fast gate <pass/fail>, judge <approved/rejected[: reason]>
- Primary: <best> -> <new> (<+/-%>)
- Secondary: alloc <..>, ns <..>, cpu <..>, mutexWait <..> (note regressions)
- Outcome & why: <why kept or discarded; what you learned for next time>
- Commit: <hash if kept>
```

## Ideas backlog (starting points, not a script)

Glass DB latency is dominated by object-storage round-trips, so the biggest wins
reduce backend operations (`MetaReads`/`MetaWrites`/`ObjReads`/`ObjWrites`):

- Collapse redundant metadata round-trips (`GetMetadata` + `SetTagsIf`) in the
  read/validate/commit path; cache more aggressively where the cache is sound.
- Batch or parallelize independent backend calls (see `internal/concurr` fanout)
  so a transaction issues fewer sequential round-trips.
- Read path: avoid promoting weak reads to strong reads unnecessarily; reuse
  already-fetched versions within a transaction.
- Commit path: reduce the number of objects written per committed key; fold
  metadata updates into existing writes.
- Locking (`internal/trans`, `internal/storage/locker.go`): cut backend ops in
  lock acquisition/release; reduce contention on shared maps
  (`internal/shard`, `internal/cache`) to improve `mutexWaitNsPerTx`.
- Allocation hot spots in `tx.go`/`internal/storage`: reuse buffers, avoid
  per-op allocations to improve `allocBytesPerTx`/`allocsPerTx`.

Prefer changes that are obviously correct. If a change makes the serializability
reasoning harder, it is probably not worth it.

## Budget & stop conditions

- Run experiments until the human-provided budget (time or count) is exhausted.
- Stop early and summarize if: the tree cannot be returned to a clean state, the
  baseline `--full` gate fails, or you have made many consecutive experiments
  with no improvement (reflect in the log and try a different area).
- At the end, write a short summary at the bottom of `log.md`: total
  experiments, kept count, baseline score -> best score.
